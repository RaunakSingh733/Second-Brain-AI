# Contributing to Second Brain AI\n\nThank you for your interest in contributing to Second Brain AI! This document provides guidelines and instructions for contributing to the project.\n\n## Code of Conduct\n\n- Be respectful and inclusive\n- Provide constructive feedback\n- Focus on code quality and functionality\n- Report security issues privately\n\n## How to Contribute\n\n### 1. Report Bugs\n\nBefore reporting a bug:\n- Check if it's already reported in [Issues](https://github.com/RaunakSingh733/Second-Brain-AI/issues)\n- Try to reproduce it with the latest code\n\nWhen reporting:\n- Provide a clear description\n- Include steps to reproduce\n- Share error messages and logs\n- Mention your environment (OS, Python, Node.js versions)\n\n### 2. Suggest Features\n\nBefore suggesting:\n- Check existing issues and PRs\n- Consider if it aligns with project goals\n\nWhen suggesting:\n- Describe the feature clearly\n- Explain the use case\n- Discuss potential implementation\n- Consider alternatives\n\n### 3. Submit Pull Requests\n\n#### Setup Development Environment\n\n```bash\n# Clone your fork\ngit clone https://github.com/YOUR_USERNAME/Second-Brain-AI.git\ncd Second-Brain-AI\n\n# Add upstream remote\ngit remote add upstream https://github.com/RaunakSingh733/Second-Brain-AI.git\n\n# Create a feature branch\ngit checkout -b feature/your-feature-name\n```\n\n#### Development Guidelines\n\n**Backend (Python)**\n```bash\ncd backend\npython -m venv venv\nsource venv/bin/activate  # or venv\\Scripts\\activate on Windows\npip install -r requirements.txt\n```\n\n**Frontend (TypeScript/React)**\n```bash\ncd frontend\nnpm install\nnpm run dev\n```\n\n#### Code Style\n\n**Python**\n- Follow PEP 8 guidelines\n- Use type hints\n- Include docstrings for functions/classes\n- Maximum line length: 100 characters\n\n**TypeScript/JavaScript**\n- Use TypeScript for type safety\n- Follow ESLint configuration\n- Use meaningful variable names\n- Add comments for complex logic\n\n**General**\n- Write clean, readable code\n- Keep functions small and focused\n- Avoid code duplication\n- Test your changes\n\n#### Commit Messages\n\nFollow conventional commits format:\n```\n<type>: <subject>\n\n<body>\n\n<footer>\n```\n\nTypes:\n- `feat`: A new feature\n- `fix`: A bug fix\n- `docs`: Documentation changes\n- `style`: Code style changes (formatting, etc.)\n- `refactor`: Code refactoring without feature changes\n- `test`: Adding or updating tests\n- `chore`: Dependency or configuration changes\n\nExample:\n```\nfeat: Add hybrid search endpoint\n\nImplements combined vector and keyword search with\nconfigurable weights. Includes proper error handling\nand validation.\n\nFixes #123\n```\n\n#### Pull Request Process\n\n1. **Update and rebase**\n   ```bash\n   git fetch upstream\n   git rebase upstream/main\n   ```\n\n2. **Push your branch**\n   ```bash\n   git push origin feature/your-feature-name\n   ```\n\n3. **Create Pull Request**\n   - Use a clear title\n   - Reference related issues\n   - Describe changes thoroughly\n   - Include screenshots/demos if applicable\n\n4. **Respond to Reviews**\n   - Make requested changes\n   - Update your PR with new commits\n   - Discuss concerns respectfully\n\n### 4. Improve Documentation\n\nDocumentation improvements are always welcome!\n\n- Fix typos and unclear explanations\n- Add examples and clarifications\n- Update outdated information\n- Add troubleshooting tips\n\n---\n\n## Project Structure\n\n```\nSecond-Brain-AI/\n├── frontend/          # Next.js React application\n│   ├── app/           # App pages and routing\n│   ├── components/    # React components\n│   ├── services/      # API client\n│   └── types/         # TypeScript interfaces\n├── backend/           # FastAPI Python backend\n│   ├── api/           # API routes\n│   ├── services/      # Business logic\n│   ├── embeddings/    # Embedding services\n│   ├── vector_db/     # ChromaDB wrapper\n│   ├── ai/            # Ollama integration\n│   ├── database/      # SQLite wrapper\n│   └── parsers/       # Document parsing\n└── docs/              # Documentation\n```\n\n---\n\n## Testing\n\n### Backend Testing\n\n```bash\n# Run backend tests\ncd backend\npython -m pytest tests/\n\n# Test with coverage\npython -m pytest tests/ --cov\n```\n\n### Frontend Testing\n\n```bash\n# Run frontend tests\ncd frontend\nnpm run test\n\n# Test with coverage\nnpm run test:coverage\n```\n\n### Manual Testing\n\nBefore submitting:\n1. Test your feature locally\n2. Test edge cases\n3. Verify error handling\n4. Check performance if applicable\n\n---\n\n## Development Phases\n\n### Phase 1: Foundation ✅\n- Basic project structure\n- File upload handling\n- Simple chat interface\n\n### Phase 2: Core Features ✅\n- Hybrid search\n- Chat memory\n- Auto-summarization\n- Source citations\n\n### Phase 3: Enhancements (Planned)\n- OCR support\n- Flashcard generation\n- Advanced UI/UX\n- Document organization\n\n### Phase 4: Advanced Features (Planned)\n- Knowledge graph\n- Multi-user support\n- API authentication\n- Advanced search\n\n---\n\n## Issues and Discussions\n\n- **Bugs**: Report in [Issues](https://github.com/RaunakSingh733/Second-Brain-AI/issues)\n- **Features**: Discuss in [Discussions](https://github.com/RaunakSingh733/Second-Brain-AI/discussions)\n- **Security**: Email privately instead of creating an issue\n\n---\n\n## Getting Help\n\n- Read [README.md](./README.md) for project overview\n- Check [SETUP.md](./SETUP.md) for setup help\n- Review documentation files\n- Ask in GitHub Discussions\n- Check existing issues and PRs\n\n---\n\n## Recognition\n\nContributors will be:\n- Listed in CONTRIBUTORS file\n- Acknowledged in release notes\n- Credited in documentation\n\n---\n\n## License\n\nBy contributing, you agree that your contributions will be licensed under the MIT License.\n\n---\n\n## Questions?\n\nFeel free to ask in:\n- GitHub Issues (for bugs/features)\n- GitHub Discussions (for questions)\n- Comments on PRs\n\nThank you for contributing! 🚀\n
